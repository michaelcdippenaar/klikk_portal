2. Why the portal's table needed more

The portal isn't rendering 3 rows. It's rendering 20,503 bank transactions across 11 accounts, with seven concerns the bare HTML doesn't address:

  • Virtual scroll — putting 20k <tr> in the DOM kills the browser. Solution: render only the ~15 visible rows, absolute-position them, recompute on scroll. That's TanStack useVirtualizer.
  • Sort, filter, pagination — bare HTML doesn't sort. You need a row-model engine (TanStack Table).
  • Sticky thead — the header has to stay visible when you scroll 20k rows. position: sticky on <thead>.
  • Column alignment — virtual rows are position: absolute, which means they don't influence column widths. The browser sizes columns from <thead> alone and you get header-left, data-clustered drift. Solution: table-layout: fixed + <colgroup> with explicit widths.
  • Dropdown over thead — when an Account-selector dropdown opens, sticky thead at z-index: 2 covers it. The dropdown is teleported to <body> by the popover library. Its z-index has to be higher (we use a 1000+ token scale).
  • Accessibility — keyboard sort (Tab + Enter on <th>), aria-controls linking filters → table, aria-live on empty states, screen-reader-friendly result count.
  • Slot contract — Vue passes the row data to a slot template, but is it row or row.original? Get it wrong, the whole table renders nothing. Twice in one session.
The honest take: the "drama" wasn't HTML. The four tags work perfectly. The drama was making them work at the scale + interactivity + a11y bar of a finance ops console. Most of the time we were paying for one of those seven concerns, not for the <table> itself.
